ЗАДАНИЕ 01
Автоматическая сортировка файлов
BEGINNER
Дедлайн: Неделя 2 12 баллов
СТАТУС
🎯 Цель проекта

Разработать автоматическую систему сортировки файлов, которая мониторит указанную директорию в реальном времени и раскладывает новые файлы по папкам согласно настраиваемым правилам. Решение должно работать как фоновый демон с полным логированием операций.

💼 Бизнес-кейс

Отдел финансового документооборота компании получает ежедневно 200–500 файлов разных форматов на общий сетевой диск. Бухгалтеры тратят 1.5–2 часа в день на ручное перекладывание файлов по папкам. Требуется автоматизировать процесс: PDF-счета → в /invoices, XLS-отчёты → в /reports, JPEG-сканы → в /scans, и т.д.

📋 Техническое задание
Настройка окружения: создать виртуальное окружение venv, установить watchdog, python-dotenv. Зафиксировать зависимости в requirements.txt с точными версиями через pip freeze.
Конфигурация через .env: вынести SOURCE_DIR, LOG_DIR, LOG_LEVEL, CONFIG_PATH в .env файл. Создать .env.example с комментариями для каждой переменной. Загрузка через python-dotenv.
Правила сортировки в JSON: разработать структуру rules.json с маппингом расширений → папки. Поддержка wildcards (*.tmp, *.bak), групп расширений (images: [jpg, png, gif]), правил по маске имени файла.
Загрузка и валидация конфига: модуль config.py читает rules.json, валидирует структуру через dataclasses или Pydantic. При ошибке конфига — понятное сообщение с указанием строки и поля.
Мониторинг директории через watchdog.FileSystemEventHandler — реагировать на FileCreatedEvent и FileMovedEvent. Игнорировать временные файлы (*.tmp, ~*). Добавить задержку 0.5s перед обработкой — файл может ещё писаться.
Логика сортировки: функция sort_file(path) определяет целевую папку по правилам. Приоритет правил: точное совпадение имени > маска > расширение > группа > default. Возвращает путь назначения или None если нет подходящего правила.
Создание целевых папок: перед перемещением убедиться что папка назначения существует. pathlib.Path.mkdir(parents=True, exist_ok=True). Не создавать папки в dry-run режиме.
Обработка конфликтов имён: при совпадении имён файлов добавлять timestamp к имени (_20240115_143022), не перезаписывать. Функция resolve_conflict(target_path) → уникальный Path.
Безопасное перемещение: обернуть shutil.move() в try/except PermissionError, FileNotFoundError, OSError. При PermissionError — retry 3 раза с задержкой 1s. Если после 3 попыток ошибка — записать в failed_files.log.
Логирование операций: каждая операция перемещения — в файл sort.log с RotatingFileHandler (max 5MB × 3 файла). Формат: timestamp · уровень · файл → папка · размер · время операции в мс.
Статистика сессии: класс SessionStats собирает: количество перемещённых файлов, суммарный размер, разбивку по категориям, список ошибок, время работы. Потокобезопасный (threading.Lock).
Graceful shutdown: перехватить SIGTERM и KeyboardInterrupt через signal.signal(). При остановке — дождаться завершения текущей операции, распечатать итоговый отчёт сессии через rich.console или tabulate.
CLI-интерфейс через argparse: --source (директория), --config (путь к rules.json), --dry-run (показать что сделает без выполнения), --daemon (запустить как фоновый процесс), --verbose (подробный вывод).
Dry-run режим: запустить полную логику определения целевой папки и разрешения конфликтов, но без реального перемещения. Вывести красивую таблицу в терминал: файл | будет перемещён в | размер.
Обработка уже существующих файлов: при запуске с флагом --initial-scan — пройти по всем существующим файлам в source и отсортировать их. Защита от повторной сортировки уже отсортированных файлов через SQLite-журнал.
Undo-функция: каждое перемещение записывается в undo_log.json {timestamp, src, dst}. Команда --undo N отменяет последние N операций в обратном порядке. Защита от undo если файл уже перемещён кем-то ещё.
Юнит-тесты: pytest-тесты для sort_file(), resolve_conflict(), config validation, stats. Использовать tmp_path fixture для изолированных тестов файловой системы. Coverage > 70% для модулей sorter.py и config.py.
Dockerfile: многоэтапная сборка (python:3.12-slim). Bind-mount для мониторируемой папки через volume. docker-compose.yml с restart: always, healthcheck через проверку PID файла, монтирование logs/ как named volume.
GitHub Actions CI: workflow .github/workflows/ci.yml — запуск pytest, flake8, mypy при push в main. Badge статуса в README. Публикация coverage report как artifact.
Документация README.md: описание проекта, схема rules.json с примерами, Quick Start за 5 минут, раздел Docker deployment, таблица всех CLI-флагов, примеры типичных конфигураций (для Downloads, для рабочего стола, для серверной директории).

⚠️ Типичные ошибки
❌ ЧАСТЫЕ ОШИБКИ НАЧИНАЮЩИХ

  • Не обрабатывать PermissionError при доступе к занятым файлам
  • Не добавлять задержку (time.sleep(0.5)) перед перемещением — файл может ещё писаться
  • Хардкодить пути в коде — всё должно быть в .env или config.json
  • Не проверять существование целевой папки перед shutil.move()

  • Написать Docker-контейнер с bind-mount для мониторируемой папки

ЗАДАНИЕ 02
Скрипт резервного копирования данных
INTERMEDIATE
Дедлайн: Неделя 2 12 баллов
СТАТУС
🎯 Цель проекта

Создать корпоративный скрипт инкрементального резервного копирования данных с поддержкой локального и удалённого хранения (FTP/SFTP), сжатием архивов, ротацией старых копий и уведомлениями о результате.

💼 Бизнес-кейс

Небольшое предприятие хранит базы 1С, Excel-таблицы и документы на локальном сервере. Системный администратор делает бэкапы вручную раз в неделю — это ненадёжно. Нужна автоматизированная система: ежедневный бэкап в 03:00, хранение 7 последних копий, сжатие ZIP, уведомление директора в Telegram об успехе/ошибке.

📋 Техническое задание
Настройка окружения: создать venv, установить зависимости. Структура проекта с разделением на модули: backup.py, archiver.py, notifier.py, reporter.py, config.py, scheduler.py.
Конфигурация через .env: SOURCE_DIRS (список через запятую), BACKUP_DIR, MAX_BACKUPS, COMPRESSION_LEVEL (0-9), TELEGRAM_TOKEN, TELEGRAM_CHAT_ID, SCHEDULE_TIME, LOG_LEVEL. Валидация всех параметров при старте.
Сканирование источника: рекурсивный обход SOURCE_DIR через pathlib. Для каждого файла получить: путь, размер, mtime, SHA256-хеш (hashlib). Построить словарь {relative_path: {size, mtime, hash}}.
Инкрементальный алгоритм: загрузить manifest последнего бэкапа из JSON. Сравнить с текущим сканированием. Отобрать файлы: новые (нет в manifest), изменённые (hash отличается), удалённые (есть в manifest, нет в источнике).
Полный бэкап (--full): игнорировать manifest, копировать все файлы. Использовать при первом запуске или при повреждении manifest. Флаг --full через argparse.
ZIP-архивирование: zipfile.ZipFile с компрессией ZIP_DEFLATED. Имя архива: backup_YYYYMMDD_HHMMSS.zip. Добавлять файлы с сохранением относительных путей. Прогресс-бар через tqdm или rich.progress.
Верификация архива: после создания проверить целостность ZIP через zipfile.testzip(). При ошибке — удалить повреждённый архив, записать в лог, отправить Telegram-алерт.
Обновление manifest: после успешного архивирования сохранить manifest.json в директории бэкапа. Формат: {created_at, source_dir, files_count, total_size, file_list: {...}}.
Ротация архивов: после сохранения нового архива — получить список всех backup_*.zip отсортированный по дате. Удалить архивы старше MAX_BACKUPS. Логировать каждое удаление с размером освобождённого места.
Проверка свободного места: перед началом архивирования вычислить размер нового архива (размер источника × 0.7 как оценка). Проверить shutil.disk_usage(BACKUP_DIR). Если менее 10% диска свободно — остановить и отправить алерт.
Генерация отчёта: файл backup_report_YYYYMMDD.txt — время начала/конца, длительность, размер архива, количество файлов (новых/изменённых/удалённых), скорость записи MB/s, топ-10 самых больших файлов, список ошибок.
Telegram-уведомления: при успехе — сообщение с emoji, кратким summary (размер, файлов, время). При ошибке — отдельное сообщение с traceback (первые 3000 символов). Функция send_telegram(text, parse_mode='HTML').
Планировщик задач: schedule library — запускать бэкап по расписанию из SCHEDULE_TIME (формат HH:MM). Daemon loop с sleep(1). При ручном запуске без --daemon — выполнить бэкап сразу и завершиться.
Восстановление из бэкапа: команда --restore ARCHIVE_PATH [--target DIR] — распаковать архив в указанную директорию. С подтверждением если целевая директория не пуста. Проверка целостности перед распаковкой.
Логирование: RotatingFileHandler (10MB × 5 файлов). Отдельный лог для ошибок (ERROR и выше). Формат: ISO timestamp · PID · level · module · message. Логировать начало/конец каждой операции с длительностью.
Обработка ошибок и retry: декоратор @retry(attempts=3, delay=5) для сетевых операций (Telegram API). При PermissionError на файл — пропустить файл, залогировать, продолжить. Не прерывать весь бэкап из-за одного файла.
Тесты: pytest с tmp_path. Тестировать: инкрементальный алгоритм (create/modify/delete detection), ротацию (создать N+1 архивов → проверить что N осталось), верификацию (намеренно повредить ZIP → поймать ошибку).
Dockerfile: FROM python:3.12-slim. Volumes для source и backup директорий. Cron внутри контейнера или schedule daemon. HEALTHCHECK через проверку last_backup_timestamp из manifest.
docker-compose.yml: сервис backup-system с restart: always, bind-mounts для источника и хранилища, .env через env_file, healthcheck. Опциональный сервис для просмотра логов через dozzle.
README и документация: Quick Start, описание всех переменных .env с примерами, схема инкрементального алгоритма в Mermaid, инструкция по восстановлению данных, FAQ по типичным ошибкам (нет места, Telegram недоступен, повреждён manifest).

ЗАДАНИЕ 03
Генератор Excel-отчётов с диаграммами
INTERMEDIATE
Дедлайн: Неделя 3 13 баллов
🎯 Цель проекта

Создать систему автоматической генерации ежедневных Excel-отчётов из CSV-данных или базы данных. Отчёт должен включать стилизованные таблицы, сводные данные, диаграммы, условное форматирование и автоматически отправляться по email.

💼 Бизнес-кейс

Руководство интернет-магазина каждое утро ожидает Excel-отчёт по продажам за вчера: сводка по категориям товаров, топ-10 позиций, сравнение с прошлой неделей, диаграмма тренда. Аналитик тратит 40 минут на сборку вручную. Нужен скрипт, который читает CSV-выгрузку из CRM и генерирует готовый отчёт за 10 секунд.

📋 Техническое задание
Настройка проекта: структура модулей data_processor.py, excel_builder.py, emailer.py, config.py, scheduler.py. Типизация через dataclasses для ReportConfig, SalesData, ReportResult.
Конфигурация: .env с CSV_PATH или DB_CONNECTION, OUTPUT_DIR, EMAIL_RECIPIENTS (список), SMTP_HOST/PORT/USER/PASS, SCHEDULE_TIME, REPORT_PERIOD_DAYS. Валидация Pydantic.
Чтение данных из CSV: pandas читает CSV с продажами. Автоматическое определение кодировки через chardet. Валидация обязательных колонок с понятными сообщениями. Приведение типов дат через pd.to_datetime.
Альтернативный источник — SQLite/PostgreSQL: опциональная загрузка через pd.read_sql(). Параметризованный SQL-запрос с диапазоном дат. Закрытие соединения через context manager.
Агрегация данных: groupby по категориям товаров → сумма выручки, количество. pivot_table по дням → таблица продаж. nlargest(10) для топ-позиций. pct_change() для сравнения с предыдущим периодом.
Вычисление KPI: итоговая выручка, средний чек, количество заказов, конверсия по категориям, самый продаваемый час дня. Оформить как датакласс KPIReport для передачи в excel_builder.
Структура Excel-файла: 4 листа — «Сводка» (KPI), «По категориям», «Топ-10 товаров», «Тренд по дням». openpyxl.Workbook с именованными листами. Заголовок с датой генерации.
Стилизация заголовков: PatternFill для шапки таблицы (тёмный фон), Font(bold=True, color='FFFFFF'), Alignment(horizontal='center'). Ширина колонок через column_dimensions[col].width = max_length.
Условное форматирование: ColorScaleRule для столбца «Изменение %» — красный (отрицательные) → белый (0) → зелёный (положительные). DataBar для столбца выручки. Rule через openpyxl.formatting.rule.
Числовые форматы: суммы в формате #,##0.00 ₽, проценты в 0.0%, даты в DD.MM.YYYY. Применять через cell.number_format. Выравнивание: числа по правому краю, текст по левому.
Заморозка строк и фильтры: freeze_panes='A2' для фиксации заголовка. auto_filter.ref для включения фильтров на всех таблицах. Сортировка данных перед записью.

Батчинг и rate limiting: BATCH_SIZE (из .env) — количество писем в пачке. BATCH_DELAY_SECONDS — пауза между пачками. Внутри пачки: задержка random(0.5, 2) между письмами. Не превышать лимиты провайдера (Gmail: 500/день, 100/час).
Retry для неудавшихся отправок: при SMTPException — повторить 3 раза с задержкой 5s. После 3 неудач — добавить в failed_queue.csv. Команда --retry-failed перечитывает failed_queue и пробует повторно.
Трекинг отправки: delivery_log.csv с колонками: email, status (sent/failed/skipped), timestamp, error_message, smtp_response_code. Обновляется в реальном времени во время рассылки. Не перезаписывать — append mode.
Трекинг открытий (pixel tracking): встроить img src="https://your-server.com/track/{{ tracking_id }}.gif" в HTML-шаблон. Простой Flask endpoint логирует открытие. Опционально: SQLite для хранения открытий по tracking_id → email.
Unsubscribe ссылка: в шаблоне — ссылка «Отписаться» с уникальным токеном. Flask endpoint /unsubscribe/<token> добавляет email в unsubscribe_list.csv. Простая страница подтверждения отписки.
Итоговый отчёт доставки: summary.txt и summary.xlsx — статистика: всего отправлено, успешно, ошибок, пропущено (unsubscribed/invalid). Группировка ошибок по типам. Отправить summary на ADMIN_EMAIL по завершении рассылки.
Dry-run режим: --dry-run — вывести список получателей и превью письма без реальной отправки. Показать отрендеренный HTML первого письма в браузере (webbrowser.open). Подсчитать время рассылки с учётом батчинга.
Планировщик рассылки: schedule library — запустить рассылку в указанное время (SEND_AT из .env). Поддержка отложенного старта. Команда --now для немедленного запуска. Проверка расписания через --schedule-info.
Прогресс в реальном времени: rich.progress или tqdm — прогресс-бар с: количество отправленных / всего, скорость (писем/мин), ожидаемое время до конца, последний отправленный email. Обновление каждую секунду.
Тесты: pytest с MockSMTP. Тестировать: валидацию email, фильтрацию unsubscribed, рендеринг шаблона, формирование MIME, батчинг (проверить что паузы соблюдаются через mock time). Coverage > 65%.
README: схема CSV-файла с примером данных, инструкция по настройке Gmail App Password, примеры использования всех CLI-флагов, описание Jinja2 переменных шаблона, FAQ по частым ошибкам SMTP.

Создание standalone .exe-программы
INTERMEDIATE
Дедлайн: Неделя 6 10 баллов
🎯 Цель проекта

Упаковать один из предыдущих скриптов (задание 1, 2 или 3 на выбор) в standalone Windows .exe приложение с GUI-интерфейсом через tkinter, иконкой, настройками через INI-файл и автозапуском через реестр Windows.

📋 Техническое задание
Выбор базового скрипта: взять один из заданий 1, 2 или 3. Рефакторинг для GUI: разделить логику (business logic) и UI слои. Логика не должна зависеть от tkinter — тестируемая независимо.
Главное окно tkinter: Tk() с заданным размером и минимальным размером. Иконка окна через iconbitmap(). Заголовок с версией приложения. Обработка закрытия окна через protocol('WM_DELETE_WINDOW') — корректная остановка фоновых потоков.
Макет интерфейса: ttk.Notebook с вкладками (Управление, Настройки, Логи). Использовать ttk для современного вида (ttk.Button, ttk.Entry, ttk.Combobox). Frame с grid layout для выравнивания элементов. Адаптивная растяжка через columnconfigure/rowconfigure.
Панель управления: кнопки «Запустить», «Остановить», «Пауза». StringVar для статуса («Ожидание», «Работает», «Ошибка»). Label с цветным индикатором статуса. ProgressBar (ttk.Progressbar) для отображения прогресса операции.
Лог-панель: ScrolledText (readonly) для отображения логов в реальном времени. Цветовая кодировка: ERROR красным, WARNING жёлтым, INFO белым через tag_configure. Кнопка «Очистить лог». Автоскролл к последней строке.
Вкладка Настройки: поля ввода для всех параметров конфига. ttk.Entry для строковых значений, ttk.Spinbox для числовых, ttk.Checkbutton для bool. Кнопки «Сохранить» и «Сбросить к умолчаниям». Валидация при сохранении с подсветкой ошибочных полей.
Хранение настроек в INI: configparser сохраняет в app_config.ini рядом с .exe. Секции: [General], [Paths], [Network], [Notifications]. Загрузка при старте с fallback на DEFAULT значения. Кнопка «Открыть папку с конфигом» через os.startfile.
Фоновые потоки: вся бизнес-логика в threading.Thread(daemon=True). Остановка через threading.Event(). Передача результатов в GUI через Queue + root.after(100, check_queue). Запрет запуска если поток уже работает (кнопка disabled).
System Tray через pystray: Icon в трее с меню (Открыть, Запустить/Стоп, Выход). При закрытии главного окна — сворачивать в трей, не завершать. Уведомления через plyer.notification или Windows Toast при завершении задачи.
Диалоги выбора файлов: filedialog.askdirectory() для выбора папки, askopenfilename() для файлов. Валидировать выбранные пути. Отображать короткую версию пути (последние N символов) если слишком длинный.
Иконка приложения: создать app.ico (16x16, 32x32, 48x48 в одном файле) через Pillow или онлайн-конвертер. Использовать в заголовке окна, трее, и как иконку .exe через PyInstaller --icon параметр.
PyInstaller spec-файл: создать app.spec вместо команды в командной строке. Указать: a.datas для включения шаблонов/иконок, hiddenimports для неочевидных зависимостей, excludes для уменьшения размера. Проверить размер .exe — цель < 50MB.
Включение ресурсов: sys._MEIPASS для доступа к bundled-файлам при запуске из .exe. Функция resource_path(relative_path) → абсолютный путь (работает и в dev, и в .exe). Проверить что templates/, icons/, config/ доступны.
Автозапуск с Windows: функция toggle_autostart(enable: bool). winreg.OpenKey(HKEY_CURRENT_USER, Run). SetValueEx при включении, DeleteValue при выключении. Checkbox в настройках для управления. Показывать текущий статус при старте.
Обработка ошибок в GUI: messagebox.showerror() для критических ошибок. Не давать приложению упасть с uncaught exception — sys.excepthook перехватывает и показывает диалог с копированием traceback. Кнопка «Отправить отчёт об ошибке» (копировать в clipboard).
Логирование в файл: RotatingFileHandler в %APPDATA%/AppName/logs/. Отображать путь к логам в настройках. Кнопка «Открыть лог-файл» через os.startfile. Уровень логирования — переключатель в настройках (DEBUG/INFO/WARNING).
О программе: диалог Help → About с: иконка, название, версия, Python-версия, список библиотек. Ссылка на GitHub. Кнопка «Проверить обновления» (заглушка или реальный запрос к GitHub Releases API).
Тестирование .exe: запустить собранный .exe на чистой виртуальной машине без Python. Проверить все функции: настройки сохраняются, логи пишутся, трей работает, автозапуск регистрируется. Задокументировать тест-кейсы в README.
NSIS installer (опционально): установщик installer.nsi — выбор директории, ярлык на рабочем столе, запись в реестр Add/Remove Programs, деинсталлятор. Сборка через makensis. Readme по установке.
README и документация: скриншоты интерфейса, инструкция по сборке .exe (шаг за шагом), описание всех настроек, раздел для конечного пользователя (не программиста) по установке и использованию, известные ограничения и обходные решения.

Реализовать свой кейс. Необходимо использовать минимум 4 технологии курса и обосновать бизнес-ценность решения.

📋 Обязательные компоненты системы
Архитектурное проектирование: до написания кода — создать architecture.md с описанием компонентов, потоков данных (data flow diagram), выбором технологий и обоснованием. Diagramm через Mermaid или DrawIO. Согласование с ТЗ перед началом реализации.
Структура пакета src/: Python-пакет с __init__.py. Модули: collector.py, processor.py, reporter.py, notifier.py, scheduler.py, config.py, exceptions.py. Каждый модуль — единая ответственность. Circular imports не допускаются.
Кастомные исключения: иерархия в exceptions.py: BaseAppError → CollectorError, ProcessorError, ReporterError, NotifierError. Каждое исключение содержит context (что произошло) и suggestion (что делать). Логировать с уровнем ERROR при возникновении.
Сбор данных с retry: collector.py с декоратором @retry(tenacity). Логировать каждый запрос: URL, статус, время ответа. Кеширование сырых данных в data/raw/ — при повторном запуске не скачивать если файл свежее N минут. Параметр --force-refresh.
Обработка данных через pandas: processor.py — pipeline функций: load() → validate() → clean() → transform() → aggregate(). Каждая функция принимает и возвращает DataFrame. Логировать размер DataFrame на каждом шаге. Тест на пустой DataFrame.
Валидация данных: Pydantic модели для входящих данных или ручная валидация через assert. Проверить: обязательные поля, типы данных, диапазоны значений, отсутствие дубликатов по ключевым полям. При ошибке валидации — детальное сообщение что именно не так.
Генерация Excel-отчёта: reporter.py — многолистовой XLSX с KPI-дашбордом, детальными данными, диаграммами. Условное форматирование. Именование файла с timestamp. Сохранение в output/ с ротацией (хранить 30 последних).
Telegram-бот интеграция: notifier.py — команды /report (отправить последний XLSX в чат), /status (статус всех компонентов), /run (ручной запуск pipeline). Авторизация через whitelist. Логировать все команды с user_id.
Email-рассылка итогов: отправка Excel-отчёта на список recipients из .env. HTML-тело с краткой сводкой KPI через Jinja2. Не падать если SMTP недоступен — залогировать и отправить Telegram-алерт вместо email.
Планировщик APScheduler: scheduler.py — основной pipeline по расписанию (CRON-выражение из .env). Отдельные задания для очистки старых файлов, проверки здоровья компонентов. Persistence через SQLAlchemyJobStore для восстановления после рестарта.
Health check: endpoint или проверочный скрипт health_check.py — тестирует доступность всех внешних зависимостей (API, SMTP, Telegram). Возвращает JSON с статусом каждого компонента. Используется Docker HEALTHCHECK.
Comprehensive logging: structlog или стандартный logging с JSON-форматом для production. Ротирующие файлы с разными уровнями: app.log (INFO+), errors.log (ERROR+), debug.log (DEBUG, только в dev). Логировать start/end каждого pipeline с duration.
Конфигурация через Pydantic Settings: class Settings(BaseSettings) — автоматическая загрузка из .env с валидацией. Вложенные конфиги для каждого компонента. settings.py как единая точка конфигурации. .env.example с описанием каждой переменной.
Pytest тесты: тесты для collector (mock HTTP), processor (fixture DataFrames), reporter (проверить что XLSX создаётся и открывается), scheduler (mock time). Integration test для полного pipeline end-to-end с mock внешних зависимостей. Coverage > 60%.
docker-compose.yml: сервис app (основное приложение), опциональный bot (Telegram-бот как отдельный процесс). Volumes: ./data:/app/data, ./logs:/app/logs. env_file: .env. restart: always. depends_on для порядка запуска.
GitHub Actions CI/CD: workflow ci.yml — запуск pytest + flake8 + mypy на каждый push. workflow deploy.yml — при merge в main: SSH на VPS, git pull, docker-compose up -d --build. Secrets в GitHub Actions для SSH-ключа и .env переменных.
Деплой на VPS: пошаговый деплой на чистый Ubuntu-сервер. Установка Docker + docker-compose. Клонирование репозитория. Настройка .env. Запуск через docker-compose. Проверка health check. Настройка logrotate для лог-файлов.
Мониторинг работоспособности: cron-задание каждые 5 минут проверяет что контейнер работает и pipeline выполнился в ожидаемое время. При проблеме — отправить Telegram-алерт. Хранить метрику last_successful_run в файле/SQLite.
README.md: описание бизнес-ценности (что автоматизирует и сколько времени экономит), архитектурная схема, Quick Start за 10 минут, полная инструкция деплоя на VPS, описание всех переменных .env, API/команды бота, troubleshooting FAQ.
Финальная презентация: PRESENTATION.md или слайды с описанием: проблема → решение → архитектура → демонстрация → результаты (сколько времени/денег экономит) → дальнейшее развитие. Код-ревью по чеклисту production-ready.
